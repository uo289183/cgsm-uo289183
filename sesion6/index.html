Descripción de la práctica

En esta sexta práctica se trabaja con WebRTC (Web Real-Time Communication), la tecnología del navegador que permite comunicación multimedia en tiempo real sin necesidad de plugins adicionales. Se exploran las tres APIs principales de WebRTC: MediaStream (acceso a cámara y micrófono), RTCPeerConnection (distribución de medios entre pares) y RTCDataChannel (datos de baja latencia). Adicionalmente, se integra el flujo de vídeo WebRTC como textura en una escena Three.js/WebGL.

Para garantizar la compatibilidad entre navegadores se ha utilizado la biblioteca webrtc-adapter (instalada vía npm install webrtc-adapter), que abstrae las diferencias entre implementaciones de Chrome, Firefox y otros navegadores modernos.

Ejercicio 1 – Captura de vídeo con MediaStream (prac6-1)

La página prac6-1.html accede a la cámara del equipo utilizando la API navigator.mediaDevices.getUserMedia() con restricciones { audio: false, video: true }. La promesa devuelta se encadena con .then() para asignar el flujo capturado al atributo srcObject del elemento <video>, y con .catch() para gestionar errores como ConstraintNotSatisfiedError (la resolución solicitada no está disponible) o PermissionDeniedError (el usuario denegó el acceso). El navegador solicita permiso al usuario antes de abrir la cámara y muestra un indicador visual de que la cámara está en uso.

Ejercicio 2 – Captura de frames de vídeo (prac6-2)

Basada en prac6-1.html, esta página añade la capacidad de fotografiar un frame del vídeo y descargarlo como PNG. Se emplean dos elementos adicionales: un <canvas> oculto (con identificador photo-canvas) y un enlace <a> con atributo download="capture.png". El flujo es el siguiente:

  1. El evento canplay del elemento <video> se usa para determinar el tamaño real del vídeo capturado y ajustar el canvas al mismo aspecto (ancho fijo de 320 px, alto calculado proporcionalmente). La variable streaming evita la re-entrada en el manejador.
  2. Al pulsar el enlace Capturar, se llama a takepicture(), que dibuja el frame actual del vídeo en el canvas mediante canvas.getContext('2d').drawImage(video, …).
  3. La imagen se convierte a data URL PNG con canvas.toDataURL('image/png') y se asigna al atributo href del enlace. Al ser el atributo download el que activa la descarga, el navegador guarda directamente el fichero capture.png.

Ejercicio 3 – Llamada de vídeo con RTCPeerConnection (prac6-3)

Esta página implementa una llamada de vídeo local (los dos extremos están en la misma página) para ilustrar el funcionamiento de RTCPeerConnection. La interfaz presenta dos elementos <video> (local y remoto) y tres botones: Start, Call y Hang up.

El flujo de la llamada es:

  1. Start: captura el stream local con getUserMedia, lo asigna al vídeo local (localVideo.srcObject) y habilita el botón Call.
  2. Call: crea dos objetos RTCPeerConnection (caller y callee). Añade las pistas del stream local al peer local. Inicia la negociación con createOffer() → gotLocalDescription() → setLocalDescription / setRemoteDescription → createAnswer() → gotRemoteDescription(). El protocolo ICE (Interactive Connectivity Establishment) intercambia candidatos de conectividad entre ambos peers mediante los manejadores onicecandidate. Cuando el stream remoto llega al peer callee, gotRemoteTrack() lo asigna al vídeo remoto.
  3. Hang up: cierra ambas conexiones (.close()), las pone a null y restablece el estado de los botones.

En una aplicación real, los dos peers estarían en navegadores distintos y las descripciones multimedia (oferta/respuesta SDP) se intercambiarían mediante un servidor de señalización (por ejemplo, usando el protocolo SIP o WebSocket).

Ejercicio 4 – Integración WebRTC + WebGL (prac6-4)

Partiendo de la práctica 3-2 (textura de vídeo sobre un plano Three.js en rotación), se ha sustituido la fuente de vídeo (fichero .mp4) por el stream de la cámara web obtenido con getUserMedia. El proceso es:

  1. Se crea dinámicamente un elemento <video> (fuera del DOM) y se le asigna el stream de la cámara como srcObject.
  2. En cada frame del bucle de animación Three.js, si el vídeo tiene suficientes datos (readyState === HAVE_ENOUGH_DATA), se dibuja el frame actual sobre un canvas 2D de 640×480 px que actúa como fuente de la textura (THREE.Texture). Se establece texture.needsUpdate = true para que Three.js suba la textura actualizada a la GPU.
  3. El plano (PlaneGeometry) rota continua sobre el eje Y, mostrando la imagen de la cámara en tiempo real sobre una malla 3D.

Dificultades encontradas

La principal dificultad al trabajar con WebRTC es la política de seguridad de los navegadores modernos, que solo permiten el acceso a la cámara en contextos seguros (https:// o localhost). Al usar el servidor de desarrollo de webpack (webpack-dev-server) con localhost no existe este problema, pero habría que configurar HTTPS para servir desde una IP pública.

En prac6-3, las llamadas a setLocalDescription y setRemoteDescription en la función gotLocalDescription deben ejecutarse antes de llamar a createAnswer(). Dado que estas operaciones son asíncronas en algunos navegadores, el orden correcto del encadenamiento de promesas es crítico para evitar que ICE no consiga conectar los peers.

En prac6-4, al igual que en la práctica 5-2, la reproducción automática del vídeo puede estar bloqueada por el navegador. Sin embargo, al usar un stream de cámara directamente (en lugar de un elemento media de archivo), los navegadores permiten la reproducción automática de streams muted, por lo que el vídeo arranca sin necesidad de interacción del usuario.
